Why NixOS, finally

The chdkpano rig started as a single Pi with a couple of services and a bootstrap script — see pi-bootstrap.sh in the repo root for that earlier approach. It was the right call at that size: imperative, cheap to learn, no toolchain on the Mac to set up.

The rig as currently designed has:

  • Two WiFi radios (built-in + USB dongle)
  • Two operating modes (field-mode AP / desk-mode client) that need to switch atomically
  • hostapd, dnsmasq, avahi, Tailscale, openssh, udev, systemd unit, firewall rules
  • WiFi creds, SSH keys, captive-portal redirects, multiple SSID priorities
  • A Rust binary + a wasm bundle that have to be built somewhere

That's ~12 interacting concerns. The bash bootstrap version of this would be a 300-line file with conditionals, version checks, and brittle "did the last edit work?" state. NixOS makes all of it one git-tracked, rebuild-rollback-able artifact.

The pivot point. When you have > ~5 interacting config files spread across multiple services, the imperative bootstrap script stops being simpler than the Nix module. We crossed that line when dual-radio + specialisations + Tailscale entered the picture.

Repository layout

repo-root/ ├── chdkpano/ ← Rust workspace (server/, client/, Cargo.toml) └── nix/ ├── flake.nix ← entry point. Declares nixosConfigurations, │ packages, darwinConfigurations. ├── hosts/ │ └── chdk-pi.nix ← this Pi's specifics: hostname, SSH keys, │ WiFi credentials, AP password. ├── modules/ │ ├── chdkpano.nix ← reusable NixOS module: user, systemd unit, │ │ udev rule, firewall. │ ├── network-base.nix ← always-on: openssh, tailscale, avahi, │ │ the `networking.chdkpano` option set. │ ├── network-field.nix ← AP mode: hostapd + dnsmasq on wlan1, │ │ wpa_supplicant on wlan0. │ └── network-desk.nix ← client mode: wpa_supplicant on both radios, │ no hostapd, sshd everywhere. ├── pkgs/ │ └── chdkpano.nix ← the actual Rust + wasm derivation. ├── darwin/ │ └── builder.nix ← nix-darwin config that turns your Mac │ into a Linux builder via linux-builder VM. └── explainer.html ← you are here.

The file split is intentional: modules are reusable behaviour, hosts are specific values, and pkgs are derivations. If you ever build a second Pi, you add hosts/chdk-pi-2.nix, reuse all the modules, and the diff is whatever's actually different about that Pi.

flake.nix — entry point

Composes the host config from modules. The interesting bits:

  • Two outputs: a nixosConfiguration (the Pi's whole-system definition) and a packages.sdImage (the flashable image artifact derived from it).
  • nixpkgs-unstable because the Pi Zero 2 W appreciates a recent kernel for hostapd/dwc2/usb-wifi-driver freshness. Pin to a release branch if you want stability over recency.
  • rust-overlay input — pins the Rust nightly the leptos client needs, so the build is reproducible across Macs and CI.
  • Field mode is the default boot, desk mode is a specialisation. See Specialisations.

One subtlety — the repo layout:

repo-root/
├── chdkpano/    ← Rust workspace (Cargo.toml, server/, client/)
└── nix/         ← this flake

# in flake.nix
repoRoot = ../chdkpano;

We pass repoRoot to the derivation so it can find Cargo.toml, Cargo.lock, client/, and server/.

hosts/chdk-pi.nix — your specifics

This is the only file that needs values touched on first setup. Everything marked TODO:

FieldWhere to get it

Fine for a hands-off rig; if you want to change passwords from the device itself, flip that to true and use initialHashedPassword instead.

modules/chdkpano.nix

Wires the chdkpano package into the OS:

  1. System user + group (chdkpano:chdkpano) with plugdev + dialout supplementary groups.
  2. udev rule giving local users 0666 on every Canon USB device. This is what lets nusb actually open the camera without root. The alternative is uaccess tagging — slightly more correct, more brittle.
  3. systemd unit running the server as the chdkpano user, with light hardening (ProtectHome, ProtectSystem=strict, PrivateTmp). The DeviceAllow entries explicitly grant USB device access inside the hardened sandbox.
  4. Firewall hole for the configured port (default 3030).
If USB stops working after a rebuild, comment out the DeviceAllow / ProtectSystem lines first. systemd's device sandbox is the most likely culprit when nusb starts failing to open devices.

pkgs/chdkpano.nix — the derivation

This is the spiciest file in the scaffold. It builds two things in one derivation:

  • chdkpano-server (Rust → aarch64-linux ELF binary)
  • client/dist/ (Rust → wasm32-unknown-unknown via leptos + Tailwind CSS via the trunk build pipeline)

Both are produced by one rustPlatform.buildRustPackage invocation with a postBuild hook that runs trunk build --release --offline inside the same build sandbox.

Two specific things you'll likely have to iterate on the first time:

  1. The path dep on chdkptp_rs. server/Cargo.toml currently has chdkptp = { path = "../../../chdkptp_rs" } — Nix can't reach a path outside the source closure. Three fixes, pick one:
    • (a) Vendor chdkptp_rs/ as a sibling of the chdkpano repo and write a small wrapper derivation that symlinks both into the build sandbox.
    • (b) Publish chdkptp to crates.io and change the dep to chdkptp = "0.1". Cleanest.
    • (c) Switch to a git dep (chdkptp = { git = "...", rev = "..." }) and add an outputHashes entry in cargoLock.
  2. Trunk + tailwind in the build sandbox. trunk build --offline needs every binary it shells out to (wasm-bindgen, wasm-opt, npx for tailwind v4) to be available in nativeBuildInputs AND findable via PATH. The scaffold lists them; if any are missing, the error will mention the binary name.
Pragmatic shortcut for first iteration: skip the full Nix build and run cross build + trunk build --release on the Mac like before, scp the artifacts to a fixed path on the Pi (e.g. /var/lib/chdkpano), and set services.chdkpano.package to a pkgs.runCommand wrapper that just copies from that path. Lose some reproducibility, gain the ability to iterate on the rest of the Nix setup without fighting the build.

modules/network-base.nix

Always-on stuff regardless of mode:

  • openssh — keys only, no passwords, no root login.
  • avahi (mDNS) — publishes chdk-pi.local on every interface that has a link-local address. So you can ssh pi@chdk-pi.local from the Mac and browse to http://chdk-pi.local:3030 from any phone on the AP.
  • Tailscale — declarative on/off. Provide authKeyFile for first-boot auto-join, or run tailscale up manually after first boot. Once joined, you can ssh the Pi from anywhere with internet, no matter what WiFi it's currently on.
  • The networking.chdkpano option block — shared vocabulary (apSsid, apPassword, apInterface, clientInterface, apSubnet) used by both network-field and network-desk modules.
  • NetworkManager disabled — we use networking.wireless = wpa_supplicant directly. Mixing the two causes both to lose fights against the other.

modules/network-field.nix default

This is the default-boot mode. The Pi:

  1. Runs hostapd on wlan1 (USB dongle), broadcasting your SSID with a static IP of ${apSubnet}.1.
  2. Runs dnsmasq on the same interface, handing out DHCP leases .10–.50 and resolving every hostname to itself (helpful for captive-portal-style redirects).
  3. Keeps wpa_supplicant on wlan0 (built-in) so the Pi also joins any known SSID in range — meaning deploys still work if you carry the rig back into home WiFi range without rebooting.
  4. Binds sshd to loopback only (Tailscale is your remote-ssh channel). Strangers on the AP can't poke at port 22.
  5. Disables IP forwarding — AP clients can only reach the Pi itself, they do not route through wlan0 to your home network. Intentional isolation.
Channel choice. Hardcoded to channel 6 for 2.4 GHz. If you're in a city and your AP throughput is bad, run a wifi-analyzer app and switch to whichever of 1/6/11 is least crowded. Don't pick anything else — they overlap.

modules/network-desk.nix specialisation

Swap-target for when the rig comes back to your desk:

  1. mkForce-disables hostapd and dnsmasq.
  2. Adds wlan1 to the wpa_supplicant client list — both radios act as clients now.
  3. Removes the static IP on wlan1 (DHCP from your home AP).
  4. Unbinds sshd's listen restriction — you're inside your trusted LAN, listen on everything.

Switch into it on the Pi:

sudo /run/booted-system/specialisation/desk/bin/switch-to-configuration switch

Switch back:

sudo /run/current-system/bin/switch-to-configuration switch

Both swaps take ~3–5 seconds. No reboot. The kernel doesn't blink; systemd just stops + starts the services that changed.

darwin/builder.nix

Your Mac can't natively produce aarch64-linux binaries (different OS, different ABI). Three ways to fix that:

OptionSpeedSetup pain
QEMU binfmt emulation~30 min image builds1 line
Remote aarch64 Linux box you ssh intofastYou need the box
nix-darwin linux-builder~3–10 min~30 min one-time

linux-builder spawns a small NixOS VM on the Mac via the macOS Virtualization framework (no Docker, no UTM, no third-party hypervisor). The nix-daemon transparently offloads aarch64-linux builds to it. You write nix build .#sdImage and don't think about it.

Setup:

# on the Mac, install nix and nix-darwin
curl --proto '=https' --tlsv1.2 -sSf -L https://install.determinate.systems/nix | sh -s -- install
nix run nix-darwin -- switch --flake ./nix#mac-builder

# verify the builder VM is alive
sudo launchctl print system/org.nixos.linux-builder | head -10
nix store ping --store ssh-ng://linux-builder

After that, every subsequent nix build for a Linux target uses the VM. linux-builder persists across reboots (it's a launchd service).

The dual-radio architecture

Pi Zero 2 W has one onboard 2.4 GHz radio. Adding a USB WiFi dongle gives you a second radio. The interface naming convention:

ifaceRadioRole in field modeRole in desk mode
wlan0Built-in BCM43436wpa_supplicant client (joins home/phone WiFi if in range)wpa_supplicant client
wlan1USB donglehostapd AP — broadcasts chdkpanowpa_supplicant client
Dongle drivers. Some Realtek USB WiFi chips (RTL8821CU, RTL8822BU, etc.) need out-of-tree kernel modules. nixos-unstable ships modern kernels with most of these in-tree, but if your dongle doesn't enumerate, dmesg | grep -i usb and add the relevant driver to boot.extraModulePackages in network-base.nix. See the commented-out entries there for the common ones.

Why two radios at all? Because in field mode the Pi also wants to be a client (for Tailscale-over-internet if you're in range of a hotspot). Pi Zero 2 W's single radio can't do both modes simultaneously — hence the USB dongle.

Specialisations explained

A NixOS specialisation is essentially "a second whole-system config that shares a kernel + bootloader with the parent." Useful when two configs differ in many places but you don't want a reboot to switch. The bootloader entry lets you boot directly into a specialisation; switch-to-configuration switch can flip live. 

specialisation.desk.configuration = {
  imports = [ ./modules/network-desk.nix ];
};

What this produces in the image:

/run/current-system ← what's running right now /run/booted-system ← what booted (same as current unless you switched) /run/booted-system/specialisation/ └── desk/ ← the desk variant, ready to activate ├── bin/switch-to-configuration └── ...

Flip mode:

# field → desk
sudo /run/booted-system/specialisation/desk/bin/switch-to-configuration switch

# desk → field (i.e. go back to the parent / default)
sudo /run/current-system/bin/switch-to-configuration switch

Both are atomic. systemd stops services that were removed, starts services that were added, reloads units that changed. Failures are rolled back automatically by the activation script.

Build + flash flow (first time)

  1. Edit nix/hosts/chdk-pi.nix — fill in TODOs (SSH key, WiFi password, AP password, timezone).
  2. cd nix && nix build .#sdImage
    First build is slow (downloads aarch64 toolchain via the linux-builder VM, builds everything from scratch). Subsequent builds hit the substituter cache for almost everything.
  3. Result is a zstd-compressed image: 
    ls -lh result/sd-image/
# nixos-sd-image-24.11-aarch64-linux.img.zst
  4. Flash it: 
    # find the SD card (will be /dev/diskN — be careful!)
diskutil list
diskutil unmountDisk /dev/diskN
zstd -dc result/sd-image/*.img.zst | sudo dd of=/dev/rdiskN bs=4m status=progress
sync && diskutil eject /dev/diskN
  5. Boot the Pi. First boot takes ~60s while the rootfs grows to fill the card. mDNS comes up around the 90s mark.
  6. From the Mac: ssh pi@chdk-pi.local — keys-only, no password prompt. Browse to http://chdk-pi.local:3030 from any device on the AP.

Iterating after first flash

You almost never want to reflash. Two faster paths exist:

Build on Mac, push activation to Pi

nixos-rebuild switch --flake ./nix#chdk-pi \
    --target-host pi@chdk-pi.local \
    --build-host localhost \
    --use-remote-sudo

Builds the new system closure on the Mac (using linux-builder), rsyncs the diff to the Pi, runs switch-to-configuration on the Pi. Usually completes in ~30 seconds for code-only changes because the substituter cache handles the unchanged closure.

Build on Pi (slow but no Mac dependency)

ssh pi@chdk-pi.local 'sudo nixos-rebuild switch --flake github:you/chdkpano#chdk-pi'

The Pi pulls the flake from GitHub and builds locally. The 512 MB RAM becomes a serious problem here — image-scale rebuilds will OOM. Code-only rebuilds usually fit. Add a swap file (1–2 GB) before relying on this.

Just the chdkpano binary

nix build ./nix#chdkpano                       # build the package only
nix copy --to ssh://pi@chdk-pi.local ./result  # push to Pi
ssh pi@chdk-pi.local 'sudo systemctl restart chdkpano'

Equivalent to the old ./deploy.sh workflow but reproducible.

Secrets & WiFi credentials

Tiers, by escalating paranoia:

TierMechanismUse when
0Cleartext in the flake (psk = "...")Personal rig, private flake repo. Simplest.
1Precomputed PSK (pskRaw = "<hash>" via wpa_passphrase)Want to push the flake to a public repo without leaking the literal password. Cracking the hash is still trivial offline.
2Runtime file (environmentFile = "/etc/wpa.env")Secret stays out of the Nix store. File goes on the SD card via a separate channel (manual scp).
3sops-nix / agenix — encrypted in the flake, decrypted on the Pi using a key burned into the imageMulti-Pi fleet, CI builds the image, want git-safe encrypted secrets.

The scaffold uses tier 0. Upgrading is a one-file change once you outgrow it.

Common operations

Add a package to the Pi (e.g. tcpdump)

# in hosts/chdk-pi.nix
environment.systemPackages = with pkgs; [ tcpdump strace btop ];
# then on the Mac
nixos-rebuild switch --flake ./nix#chdk-pi --target-host pi@chdk-pi.local

Rotate Tailscale auth key

# generate a new key at https://login.tailscale.com/admin/settings/keys
# on the Pi:
echo 'tskey-auth-NEWKEY' | sudo tee /etc/tailscale-auth.key
sudo systemctl restart tailscaled

Switch from field to desk mode and back

sudo /run/booted-system/specialisation/desk/bin/switch-to-configuration switch
# do desk things, then later:
sudo /run/current-system/bin/switch-to-configuration switch

Roll back a bad rebuild

# at the Pi shell
sudo nixos-rebuild switch --rollback
# or pick a specific generation:
sudo nix-env --profile /nix/var/nix/profiles/system --list-generations
sudo /nix/var/nix/profiles/system-N-link/bin/switch-to-configuration switch
# nuclear option: reboot, hold shift at u-boot, pick previous generation

Garbage-collect old generations on the Pi

# keep last 5
sudo nix-collect-garbage -d --delete-older-than 14d

Gotchas worth knowing in advance

  • 512 MB RAM is tight for NixOS at runtime. Add a 1–2 GB swap file on the SD card. nixos-rebuild on-device for big closures will OOM without it.
  • SD-card wear. NixOS writes the journal aggressively by default. Consider services.journald.extraConfig = "Storage=volatile"; for production rigs, or a USB SSD for the rootfs.
  • Hostapd vs NetworkManager. They fight. We disabled NM in network-base.nix; don't re-enable.
  • wpa_supplicant only sees networks you've declared. There's no captive-portal-style "let me pick a SSID from a list" UX on the device. Add new SSIDs to hosts/chdk-pi.nix and rebuild.
  • USB WiFi dongles vary wildly. Realtek RTL8188CUS-based dongles are most likely to just work. RTL8821AU/CU needs rtl8821cu module. The driver fight is the most common reason a fresh setup hangs at hostapd waiting for wlan1.
  • Linux-builder VM eats Mac resources. The default config in darwin/builder.nix reserves 4 cores + 6 GB RAM. Lower those if your Mac feels sluggish.
  • First image build downloads ~3 GB. Of nixpkgs + kernel sources + rust toolchain. After that the substituter cache handles 99% of fetches.

Glossary

flake
A directory with flake.nix + flake.lock. The lock pins every input (nixpkgs, etc.) to an exact commit so builds are reproducible across machines and time.
derivation
The "recipe" for building one thing. pkgs/chdkpano.nix returns a derivation. Nix evaluates it, downloads inputs, runs the build in a sandbox, stores the output at /nix/store/<hash>-chdkpano-0.1.0/.
module
A reusable function from { config, lib, pkgs, ... } to a partial system config. modules/chdkpano.nix is a module. Composing modules is how NixOS' "this system has these properties" definition gets built.
option
A typed knob a module exposes (lib.mkOption). services.chdkpano.port is an option. Other modules can read it via config.services.chdkpano.port and override it via services.chdkpano.port = ....
generation
A snapshot of the activated system. Every nixos-rebuild switch creates a new one and points /run/current-system at it. Old generations stay around (and bootable) until nix-collect-garbage removes them.
specialisation
A variant configuration that shares a kernel + bootloader with its parent. Useful for "two modes of the same system, swap atomically without rebooting." We use it for field vs desk.
substituter
A binary cache. cache.nixos.org stores prebuilt versions of almost every public derivation; Nix fetches from there instead of building locally when hashes match.
linux-builder
A nix-darwin-provided NixOS VM that runs on macOS. Lets your Mac transparently offload aarch64-linux / x86_64-linux builds without you ever logging into the VM.
mkForce
An override priority modifier. mkForce false beats a regular true from another module. Used in network-desk.nix to forcibly disable hostapd that network-field.nix had enabled.
